От предзаказа до готовой реплики через управляемых агентов

Мы разделяем процесс на две понятные фазы: до READY система готовит источник, после READY отдельные агенты создают и настраивают реплику. Каждый агент отвечает за свою часть, а Координатор держит общий маршрут и статус.

Контрольная точка READY

Руководство видит понятный рубеж: источник подтверждён и можно начинать реплику.

Постепенная автоматизация

Сначала заглушки, затем реальные WARP, Jira, ЕР и коннекторы без переделки ядра.

Прозрачность

У каждого действия есть статус, ответственный агент и trace по `correlation_id`.

Что продаём как идею

Проблема Перед созданием реплики нужно понять, готов ли источник, и быстро исправить недочёты.
Решение Координатор ведёт предзаказ, WARP проверяет готовность, CR Manager исправляет, ЕР-Конфигуратор стартует после READY.
Эффект Меньше ручной координации, больше трассируемости, понятная точка перехода к созданию реплики.

Главная схема

Сверху показаны агенты, которые работают в каждой фазе. Снизу — timeline статусов: сначала предзаказ до READY, затем заказ/работа по реплике после READY.

Что происходит на границе READY

Здесь лучше говорить не про endpoints, а про управленческую логику процесса.

До READY

  • Пользователь создаёт предзаказ на подготовку источника.
  • Координатор спрашивает WARP, можно ли строить реплику.
  • Если есть проблемы, CR Manager получает конкретную работу.
  • Любое “готово” перепроверяется через WARP.
  • Предзаказ не идёт дальше, пока источник не подтверждён.

После READY

  • Источник считается пригодным для создания реплики.
  • Координатор передаёт задачу ЕР-Конфигуратору.
  • ЕР-Конфигуратор создаёт заказ в другом Jira-пространстве.
  • Подбираются параметры и генерируется конфиг для ЕР.
  • Дальше подключаются агенты загрузки и публикации.

Три сценария для демонстрации

Без переключателей: каждый сценарий показан отдельно, чтобы было понятно, что именно делает система и какой ответ возвращает.

1. Источник сразу готов

  1. Пользователь создаёт предзаказ для `SRC-READY`.
  2. Координатор запускает проверку источника.
  3. WARP сразу возвращает READY.
  4. CR Manager не нужен.
  5. Координатор передаёт задачу следующей фазе.
{
  "source_id": "SRC-READY",
  "status": "READY",
  "attempts": 0,
  "cr_task_ids": [],
  "replica_started": true
}

2. Источник требует исправлений

  1. Пользователь создаёт предзаказ для `SRC-123`.
  2. WARP возвращает NOT_READY.
  3. Координатор создаёт CR-задачу.
  4. Предзаказ ждёт CR Manager.
  5. Реплика ещё не стартует.
{
  "source_id": "SRC-123",
  "status": "WAITING_CR",
  "attempts": 1,
  "cr_task_ids": ["TASK-931852627F50"],
  "replica_started": false
}

3. CR Manager завершил работу

  1. CR Manager сообщает, что задача выполнена.
  2. Координатор проверяет, что task принадлежит предзаказу.
  3. Координатор делает final-check через WARP.
  4. WARP возвращает READY.
  5. Стартует задача для ЕР-Конфигуратора.
{
  "source_id": "SRC-123",
  "status": "READY",
  "attempts": 1,
  "cr_task_ids": ["TASK-931852627F50"],
  "replica_started": true
}
Ключевая идея для согласования: мы не пытаемся сразу автоматизировать всю реплику. Сначала строим управляемый контур до READY, где понятны статусы, ответственность агентов и критерий перехода. После этого безопасно подключаем ЕР-Конфигуратора и следующих агентов.
