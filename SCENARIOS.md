# Сценарии работы Координатора

## Важная терминология

В текущем API `POST /order` технически создаёт запись с `order_id`, но по бизнес-смыслу до статуса `READY` это **предзаказ**.

Целевая логика такая: пользовательский запрос сначала попадает к **Координатору**. Координатор создаёт предзаказ в статусе `DRAFT`, назначает идентификаторы и дальше запускает нужного специализированного агента.

Предзаказ — это заявка пользователя на подготовку источника к созданию реплики.
Пользователь выбирает источник, передаёт КЭ источника, например `CM12345`, и выбирает атрибуты или поля таблицы, которые хочет заказать.

Первый специализированный шаг после создания `DRAFT`-предзаказа — **Агент управления требованиями**:

- проверяет корректность заполнения;
- проверяет выбранный атрибутный состав;
- согласует предзаказ с СДО;
- если всё корректно, возвращает Координатору результат `REQUIREMENTS_APPROVED`;
- если данных не хватает, возвращает пользователю уточняющие вопросы.

Пока предзаказ не получил статус `READY`, система работает не с самой репликой, а с источником:

- принимает пользовательский запрос и создаёт `DRAFT`-предзаказ;
- поручает Агенту управления требованиями проверить вход;
- проверяет источник через WARP;
- если готовность ниже 100%, отправляет блокирующие критерии в CR Manager;
- повторно валидирует источник;
- фиксирует trace.

Статус `READY` означает: источник прошёл официальную проверку WARP со 100% готовностью и теперь можно переходить к следующей фазе.
Если WARP возвращает меньше 100%, это всегда блокер для превращения предзаказа в заказ.

После `READY` Координатор должен не “заканчивать всё”, а превратить предзаказ в заказ и передать задачу следующему агенту — **ЕР-Координатору**.

ЕР-Координатор дальше должен:

- создать заказ в другом пространстве Jira;
- подобрать параметры реплики;
- подготовить или сгенерировать конфиг для ЕР;
- запустить дальнейший процесс создания и настройки реплики.

В текущей реализации этот следующий шаг пока представлен заглушкой `MockReplicaInitAdapter`. Позже её нужно заменить на реальный адаптер вызова ЕР-Координатора.

## Основные сущности

### Предзаказ

Предзаказ создаётся Координатором после запроса пользователя.

Пример ID:

```text
ORD-79F813B7C179
```

Да, префикс сейчас `ORD`, но бизнес-смысл на этом этапе — **предзаказ**.

Предзаказ хранит общий статус процесса подготовки источника.

### Remediation-задача CR Manager

Remediation-задача создаётся Координатором как поручение CR Manager, если WARP сказал, что источник не готов.

Пример ID:

```text
TASK-931852627F50
```

Remediation-задача — это конкретное поручение для CR Manager внутри одного предзаказа.
Сам Jira/CR-тикет создаёт уже CR Manager. Он не просто “чинит критерий руками”: он создаёт CR на доработку, запрашивает инструкции у WARP и оркестрирует инструменты или субагентов, которые закрывают критерии до 100%.

Связь такая:

```text
Предзаказ ORD-79F813B7C179
  └── remediation-задача TASK-931852627F50 для CR Manager
```

Предзаказ отвечает на вопрос:

```text
"Что пользователь хочет получить?"
```

Remediation-задача отвечает на вопрос:

```text
"Какие блокирующие критерии CR Manager должен организовать к закрытию, чтобы предзаказ мог дойти до READY?"
```

## Поля ответа API

Когда Координатор отвечает на запросы, он возвращает структуру вида:

```json
{
  "order_id": "ORD-79F813B7C179",
  "source_id": "SRC-123",
  "correlation_id": "CORR-7848a9f35f0a4f2b9c8001d328b49191",
  "status": "WAITING_CR",
  "attempts": 1,
  "cr_task_ids": ["TASK-931852627F50"],
  "replica_started": false
}
```

### `order_id`

Технический идентификатор предзаказа в Координаторе.

Пример:

```text
ORD-79F813B7C179
```

Он нужен, чтобы дальше обращаться к конкретному предзаказу:

```text
GET /order/{order_id}
POST /order/{order_id}/task-completed
```

В будущем можно переименовать это поле в `preorder_id`, если мы захотим сделать бизнес-терминологию точнее.

### `source_id`

Идентификатор источника данных.

Пример:

```text
SRC-123
```

Именно этот источник WARP проверяет на готовность.

### `correlation_id`

Сквозной идентификатор трассировки.

Пример:

```text
CORR-7848a9f35f0a4f2b9c8001d328b49191
```

Он нужен, чтобы связать события разных агентов в одну историю:

- Координатор;
- WARP;
- CR Manager;
- позже ЕР-Координатор;
- Trace Collector.

По нему можно получить trace:

```text
GET /trace/{correlation_id}
```

### `status`

Текущий статус предзаказа.

Целевые значения:

```text
DRAFT                 - предзаказ создан, но ещё не проверен
REQUIREMENTS_CHECK    - Агент требований проверяет КЭ, атрибуты и СДО
NEEDS_CLARIFICATION   - нужны уточнения от пользователя
REQUIREMENTS_APPROVED - вход корректен, можно проверять источник
VALIDATING            - Координатор запрашивает проверку WARP
WAITING_CR            - источник ниже 100%, CR Manager ведёт CR-доработку
READY                 - источник готов на 100%, можно превращать предзаказ в заказ
FAILED                - лимит попыток исчерпан или произошла критическая ошибка
```

В текущем POC пока реализована укороченная модель:

```text
CREATED     - предзаказ создан
VALIDATING  - Координатор запрашивает проверку WARP
WAITING_CR  - источник ниже 100%, CR Manager ведёт CR-доработку
READY       - источник готов на 100%, можно превращать предзаказ в заказ
FAILED      - лимит попыток исчерпан или произошла критическая ошибка
```

### `attempts`

Количество попыток исправления источника через CR Manager.

Если `attempts = 0`, значит CR Manager ещё не привлекался.

Если `attempts = 1`, значит была создана первая CR-задача.

### `cr_task_ids`

Список CR-задач, созданных внутри предзаказа.

Пример:

```json
["TASK-931852627F50"]
```

Если список пустой, значит CR Manager не понадобился.

Если в списке есть ID, значит Координатор отправлял задачи CR Manager.

### `replica_started`

Технический флаг текущей заглушки.

Сейчас он означает:

```text
Координатор вызвал MockReplicaInitAdapter после READY.
```

В целевой архитектуре это поле лучше будет заменить или расширить на что-то вроде:

```text
ep_configuration_task_started
ep_configuration_task_id
replica_phase_status
```

Потому что после `READY` должен стартовать не “готовый запуск реплики”, а задача для ЕР-Координатора.

## Сценарий 1. Источник сразу готов

### Curl-запрос

```bash
curl -s -X POST http://127.0.0.1:8000/order \
  -H "Content-Type: application/json" \
  -d '{"source_id":"SRC-READY","request":"загрузить реплику"}'
```

### Что говорит пользователь

Пользователь создаёт предзаказ:

```text
"Хочу загрузить реплику для источника SRC-READY"
```

### Что происходит внутри

Координатор:

```text
1. Создаёт предзаказ.
2. Генерирует order_id.
3. Генерирует correlation_id.
4. Ставит статус CREATED.
5. Переводит предзаказ в VALIDATING.
6. Запрашивает WARP: готов ли источник SRC-READY?
7. WARP возвращает READY.
8. Координатор переводит предзаказ в READY.
9. Координатор вызывает следующий шаг после READY.
```

В текущем POC следующий шаг — это mock `ReplicaInit`.

В целевой архитектуре следующий шаг — это задача для ЕР-Координатора:

```text
Координатор -> ЕР-Координатор:
"Источник готов. Создай заказ в другом Jira-пространстве и начни подбор параметров реплики / генерацию ЕР-конфига."
```

### Пример ответа

```json
{
  "order_id": "ORD-2E2B2C23F86B",
  "source_id": "SRC-READY",
  "correlation_id": "CORR-9ea5d97556db4b35890f278dfbeae463",
  "status": "READY",
  "attempts": 0,
  "cr_task_ids": [],
  "replica_started": true
}
```

### Как читать ответ

```text
order_id = ORD-2E2B2C23F86B
```

Координатор создал предзаказ с этим ID.

```text
source_id = SRC-READY
```

Проверялся источник `SRC-READY`.

```text
correlation_id = CORR-9ea5d97556db4b35890f278dfbeae463
```

По этому ID можно смотреть trace всей цепочки.

```text
status = READY
```

Источник сразу прошёл проверку WARP.

```text
attempts = 0
```

CR Manager не привлекался, потому что исправления не нужны.

```text
cr_task_ids = []
```

Remediation-задачи не создавались.

```text
replica_started = true
```

В POC Координатор вызвал mock следующего этапа.

В production это должно означать, что Координатор передал задачу ЕР-Координатору.

## Сценарий 2. Источник не готов, нужна remediation-задача CR Manager

### Curl-запрос

```bash
curl -s -X POST http://127.0.0.1:8000/order \
  -H "Content-Type: application/json" \
  -d '{"source_id":"SRC-123","request":"загрузить реплику"}'
```

### Что говорит пользователь

Пользователь создаёт предзаказ:

```text
"Хочу загрузить реплику для источника SRC-123"
```

### Что происходит внутри

Координатор:

```text
1. Создаёт предзаказ.
2. Генерирует order_id.
3. Генерирует correlation_id.
4. Ставит статус CREATED.
5. Переводит предзаказ в VALIDATING.
6. Запрашивает WARP: готов ли источник SRC-123?
7. WARP возвращает NOT_READY.
8. WARP сообщает failed criteria, например C1 и C3.P2.
9. Координатор увеличивает attempts до 1.
10. Координатор переводит предзаказ в WAITING_CR.
11. Координатор создаёт внутреннее поручение для CR Manager.
12. CR Manager создаёт Jira/CR-задачу и начинает remediation.
13. Координатор сохраняет связь предзаказа и поручения CR Manager.
```

### Пример ответа

```json
{
  "order_id": "ORD-79F813B7C179",
  "source_id": "SRC-123",
  "correlation_id": "CORR-7848a9f35f0a4f2b9c8001d328b49191",
  "status": "WAITING_CR",
  "attempts": 1,
  "cr_task_ids": ["TASK-931852627F50"],
  "replica_started": false
}
```

### Как читать ответ

```text
order_id = ORD-79F813B7C179
```

Координатор создал предзаказ с этим ID.

```text
source_id = SRC-123
```

Проверялся источник `SRC-123`.

```text
correlation_id = CORR-7848a9f35f0a4f2b9c8001d328b49191
```

По этому ID можно смотреть trace всей цепочки.

```text
status = WAITING_CR
```

Источник не готов. Координатор ждёт, пока CR Manager выполнит исправления.

```text
attempts = 1
```

Началась первая попытка исправления.

```text
cr_task_ids = ["TASK-931852627F50"]
```

Координатор создал поручение `TASK-931852627F50` для CR Manager.

Именно по этому поручению CR Manager создаёт Jira/CR, оркестрирует исправления и возвращает callback Координатору.

```text
replica_started = false
```

Следующий этап ещё не запущен, потому что источник не получил `READY`.

## Сценарий 3. CR Manager сообщил, что задача выполнена

### Curl-запрос

```bash
curl -s -X POST http://127.0.0.1:8000/order/ORD-79F813B7C179/task-completed \
  -H "Content-Type: application/json" \
  -d '{"cr_id":"TASK-931852627F50","status":"done","self_check_passed":true}'
```

### Что означает этот запрос

Этим запросом мы вручную имитируем CR Manager.

В реальной системе этот запрос должен отправлять сам CR Manager после того, как он:

```text
1. Создал или обновил Jira-задачу.
2. Получил remediation-инструкции от WARP.
3. Выполнил исправления через коннекторы.
4. Сделал self-check через WARP.
5. Убедился, что self-check прошёл успешно.
6. Сообщил Координатору, что CR-задача завершена.
```

### Что передаётся в URL

```text
ORD-79F813B7C179
```

Это ID предзаказа.

Он был получен в ответе сценария 2.

### Что передаётся в body

```json
{
  "cr_id": "TASK-931852627F50",
  "status": "done",
  "self_check_passed": true
}
```

Расшифровка:

```text
cr_id = TASK-931852627F50
```

Это ID remediation-поручения, которое Координатор ранее выдал CR Manager для этого предзаказа.

```text
status = done
```

CR Manager сообщает, что задача завершена.

```text
self_check_passed = true
```

CR Manager сообщает, что самопроверка через WARP прошла успешно.

### Что происходит внутри

Координатор:

```text
1. Находит предзаказ ORD-79F813B7C179.
2. Находит CR-задачу TASK-931852627F50.
3. Проверяет, что CR-задача относится именно к этому предзаказу.
4. Проверяет, что этот callback не был обработан раньше.
5. Помечает CR-задачу как COMPLETED.
6. Переводит предзаказ в VALIDATING.
7. Делает официальный final-check через WARP.
8. WARP возвращает READY.
9. Координатор переводит предзаказ в READY.
10. Координатор запускает следующий этап после READY.
```

В текущем POC следующий этап — это mock `ReplicaInit`.

В целевой архитектуре следующий этап:

```text
Координатор -> ЕР-Координатор
```

ЕР-Координатор должен создать заказ в другом Jira-пространстве и начать подбор параметров реплики / генерацию конфига для ЕР.

### Пример ответа

```json
{
  "order_id": "ORD-79F813B7C179",
  "source_id": "SRC-123",
  "correlation_id": "CORR-7848a9f35f0a4f2b9c8001d328b49191",
  "status": "READY",
  "attempts": 1,
  "cr_task_ids": ["TASK-931852627F50"],
  "replica_started": true
}
```

### Как читать ответ

```text
order_id = ORD-79F813B7C179
```

Это тот же предзаказ, который был создан во втором сценарии.

```text
source_id = SRC-123
```

Работа шла с источником `SRC-123`.

```text
correlation_id = CORR-7848a9f35f0a4f2b9c8001d328b49191
```

Trace всей цепочки можно получить по этому ID.

```text
status = READY
```

Источник прошёл финальную проверку WARP.

```text
attempts = 1
```

Была одна попытка исправления через CR Manager.

```text
cr_task_ids = ["TASK-931852627F50"]
```

В истории предзаказа осталась CR-задача, которая была создана и завершена.

```text
replica_started = true
```

В POC вызвана заглушка следующего этапа.

В production это должно означать старт задачи для ЕР-Координатора.

## Как посмотреть trace

После любого сценария можно взять `correlation_id` из ответа и выполнить:

```bash
curl -s http://127.0.0.1:8000/trace/CORR-7848a9f35f0a4f2b9c8001d328b49191
```

Trace покажет хронологию событий по предзаказу.

В текущей реализации там уже могут быть события от:

```text
coordinator
warp
cr-manager
```

В будущем туда добавятся события от:

```text
ep-configurator
init-loader
publisher
```

## Итоговая схема трёх curl-запросов

```text
1. POST /order with SRC-READY
   -> WARP returned READY
   -> preorder status READY
   -> task for next phase started

2. POST /order with SRC-123
   -> WARP returned NOT_READY
   -> preorder status WAITING_CR
   -> CR task created

3. POST /order/{order_id}/task-completed
   -> CR task marked COMPLETED
   -> WARP final-check returned READY
   -> preorder status READY
   -> task for EP Configurator should start
```
